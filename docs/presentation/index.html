01 · Business narrative

Who this is for — and why calibration beats gut feel

We predict pre-match win probability for ATP tennis. Our audience is analysts, trading desks, and operators who need repeatable, calibrated outputs — not one-off charts in a Jupyter cell. In sports and betting contexts, log loss and Brier score matter as much as accuracy: probabilities you can stake against.

02 · Notebook PoC vs MLOps product

Higher business value — by design

A notebook is great for exploration. It is not a contract with production. Below is how we turn the same idea into something you could run daily as data refreshes.

Dimension Notebook-heavy setup Our MLOps pipeline
Reproducibility Cell order, hidden state, “works on my machine” Single entrypoint python -m src.main, locked env, tests
Model promotion Which .pkl is “the” model? W&B registry + prod alias — explicit promotion
Consumption Copy-paste or re-run cells REST API + Streamlit — stakeholders self-serve
Change safety Silent breakage CI: black, flake8, pytest + ≥90% coverage
Deployment Manual, ad hoc Docker, Render, release-triggered deploy hook
Observability Print statements Structured logging, W&B runs, Render logs

03 · System architecture

One picture: config → pipeline → registry → API → UI

Everything flows from config.yaml and secrets in .env. Training is tracked in W&B; serving pulls the production artifact; Streamlit calls the same API the rest of the world could use.

04 · Production code map

Modules we ship — not slides, actual src/

Each piece has a single job. Together they form the pipeline your README documents end-to-end.

main.py

Orchestrator: W&B init, full data flow.

load → clean → validate

Ingestion, standardisation, Pandera schema gate.

features.py

Leakage-aware features + sklearn ColumnTransformer recipe.

train.py / evaluate.py

Fit pipeline, metrics, baseline comparison, W&B logging.

infer.py

Batch inference with a strict DataFrame contract.

api.py

FastAPI + Pydantic — no ML logic, only I/O contract.

app.py

Streamlit for humans: real players, surfaces, insights.

logger.py

Dual output (console + file), zero print().

notebooks/

EDA imports src/ — analysis stays aligned with prod.

05 · Real API contract

Pydantic-validated JSON — the same shape every time

GET /health exposes load status. POST /predict accepts match features and returns class + probability. OpenAPI docs are generated automatically (Swagger / ReDoc). 

POST /predict
{
  "surface": "Hard",
  "tourney_level": "G",
  "round": "F",
  "p1_rank": 1.0,
  "p2_rank": 10.0,
  "p1_hand": "R",
  "p2_hand": "L",
  "rank_diff": -9.0,
  "age_diff": -2.5,
  "ht_diff": 5.0
}
→ { "prediction": 1, "probability": 0.73 }

07 · Streamlit experience

Same model — packaged for daily use

Preset ATP players (rankings, hands, heights, flags), surface and tournament context, calibrated probability display, and match insight copy — so a non-technical stakeholder can stress-test scenarios without Postman.

Business angle

Compare favourites under different surfaces and rounds; use the UI as a “what-if” desk for pre-match narratives.

Technical angle

UI calls the deployed API — it is not a second, hidden model path.

08 · CI/CD & governance

How we avoid breaking production

CI (every PR)

Black, Flake8, pytest with coverage ≥ 90%. W&B runs offline in CI.

CD

Publishing a GitHub Release triggers the Render deploy hook — deployment is explicit.

Branches

feature/* → dev → main; protected branches, review culture.

Docker

Same image idea locally and on Render; render.yaml in repo.

10 · Model card (summary)

Transparent limits — especially for betting narratives

RandomForest on rank, surface, round, hands, age/height diff; train 2018–2020; leakage controls (P1/P2 randomisation, no post-match stats). Stronger on hard courts; no head-to-head or injury data.

Ethical positioning

Decision support, not autonomous betting. Combine outputs with domain knowledge — exactly what a serious desk would require.

11 · Continuous data — honest roadmap

What “daily use” looks like next

The product you have today is the plumbing. Plugging in daily odds feeds, scheduled retrains, and drift monitors is the natural extension — because the API and registry are already there.

  • Scheduled pipeline — nightly or weekly retrain as new ATP CSVs land.
  • Data versioning — treat incoming seasons as versioned artifacts (W&B or DVC-style).
  • Monitoring — distribution checks on ranks/surfaces; alert before silent degradation.
  • API consumers — internal dashboards or risk tools calling /predict at scale.
